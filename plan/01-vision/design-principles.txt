================================================================================
DESIGN PRINCIPLES
================================================================================

These principles govern every design decision in the system.

--------------------------------------------------------------------------------
1. DETERMINISTIC ENVELOPE, AUTONOMOUS CORE
--------------------------------------------------------------------------------

The model is not the whole system. It is the judgment engine inside a system.

    deterministic node
        -> LLM node
        -> deterministic node
        -> LLM node
        -> deterministic node

Deterministic nodes handle:
    git operations, sandbox allocation, lint/format runs, policy scans,
    secret scanners, branch push, PR creation, CI polling, notifications,
    trace recording, budget enforcement

LLM nodes handle:
    understanding the task, navigating the codebase, editing files,
    choosing tests, fixing lint failures, fixing CI failures, writing PR descriptions

Why: the model saying "I fixed it" is not evidence. Evidence is tests passed,
linters passed, CI passed, diff inspected. The deterministic envelope produces
that evidence regardless of what the model claims.

--------------------------------------------------------------------------------
2. ONE TASK, ONE SANDBOX, ONE BRANCH
--------------------------------------------------------------------------------

Every autonomous task gets:
    - its own container or VM
    - its own git branch (named agent/{task_id}-{slug})
    - its own session id and trace id
    - isolated credentials (read repo, push branch, create PR — nothing else)
    - explicit resource budgets

This prevents one agent from corrupting another and makes it trivial to
throw away failed attempts. The branch is the durable artifact.

--------------------------------------------------------------------------------
3. THE FILESYSTEM AND BRANCH ARE DURABLE STATE
--------------------------------------------------------------------------------

Do not overbuild memory in v1. Use:
    - conversation context for short-term working memory within a turn
    - a typed AgentState object for orchestration state
    - the filesystem for code and artifacts
    - git commits/branch for durable handoff to humans
    - JSONL trace logs for audit and debugging
    - Markdown files (AGENTS.md, SOUL.md, TOOLS.md, MEMORY.md) for
      persistent cross-turn context in the OpenClaw style

Vector memory is optional and additive. Better repo context and better tools
outperform generic memory for coding tasks.

--------------------------------------------------------------------------------
4. TOOL ERGONOMICS MATTER AS MUCH AS THE MODEL
--------------------------------------------------------------------------------

The agent-computer interface determines quality more than any individual prompt.

Good tools:
    - have clear, specific names
    - have short, precise descriptions with examples
    - have small typed schemas
    - return line-numbered file content
    - truncate large outputs but store full output as artifact
    - always return exit codes
    - never make irreversible changes silently

Bad tools:
    - dump enormous unstructured blobs
    - require tribal knowledge
    - make destructive changes without review
    - return success even on failure

--------------------------------------------------------------------------------
5. SUBAGENTS ARE INDEPENDENT WORKERS, NOT CHAT PARTNERS
--------------------------------------------------------------------------------

The most productive multi-agent pattern is not agents talking to each other.
It is many agents working independently on separate sandboxes.

    Task A -> Sandbox A -> Branch A
    Task B -> Sandbox B -> Branch B
    Task C -> Sandbox C -> Branch C

Subagents in this system:
    - planning agent: decomposes complex tasks, no write access
    - worker agents: implement subtasks, each in own sandbox
    - reviewer agent: reads task + diff + validation, flags issues, no write access
    - CI-fix agent: targeted loop for CI failures
    - lint-fix agent: targeted loop for lint failures

Communication between agents is via shared task DB and artifacts, not
real-time message passing.

--------------------------------------------------------------------------------
6. PROMPT LOOPS ARE FIRST-CLASS OBJECTS
--------------------------------------------------------------------------------

A prompt loop is not a scheduled shell script. It is a typed entity with:
    - a trigger (cron, webhook, issue event, PR event, manual)
    - a prompt template (with variables bound from the trigger event)
    - a target (repo, task type, environment)
    - a budget (max cost, max time, max runs/day)
    - an output policy (notify, open PR, open issue, silent)
    - a history (runs, results, errors)

Users build loops from the UI or API. The system executes them with the
same devloop blueprint as any other task.

--------------------------------------------------------------------------------
7. FAIL SAFELY AND VISIBLY
--------------------------------------------------------------------------------

A failed run should still be useful. It must leave:
    - a branch or patch if any work was done
    - a trace explaining where it stopped
    - full command outputs as artifacts
    - attempted fixes
    - the next recommended human action
    - no irreversible side effects

Never let a failure mode be silent. Every error goes to the trace page.
Every partial run produces a report.

--------------------------------------------------------------------------------
8. CHANNEL ADAPTERS DECOUPLE EVERYTHING FROM DELIVERY
--------------------------------------------------------------------------------

The agent never knows which app sent the message. Each channel adapter
normalizes inbound messages into a standard InboundMessage format:

    InboundMessage:
        channel: str
        user_id: str
        session_id: str
        content: str
        attachments: list
        metadata: dict

This means adding Telegram, WhatsApp, or Slack means writing one adapter,
not touching the agent runtime.

--------------------------------------------------------------------------------
9. CONTEXTS ARE FILES, NOT DATABASE ROWS
--------------------------------------------------------------------------------

OpenClaw's insight: persistent context should be plain Markdown files.
For each session: AGENTS.md, SOUL.md, TOOLS.md, MEMORY.md, daily log,
conversation history. These files are:
    - human-readable
    - git-committable
    - easy to inspect and modify
    - naturally versioned

The agent runtime assembles these into its context on each turn.

--------------------------------------------------------------------------------
10. MEASURE QUALITY, NOT JUST ACTIVITY
--------------------------------------------------------------------------------

Success metrics must include outcomes, not just executions:
    - PR accepted rate (human merged without changes)
    - PR merged rate
    - review change requests
    - revert rate
    - post-merge incident rate

Activity metrics (tasks run, tool calls, cost) are necessary but not sufficient.
Build an eval suite from day one. Add production trace sampling by v2.
