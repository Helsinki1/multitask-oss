================================================================================
CONTEXT PIPELINE
================================================================================

OVERVIEW
--------------------------------------------------------------------------------

Context quality determines agent quality more than any single prompt tweak.
The context pipeline: collect -> rank -> compress -> cite -> insert.

Never concatenate everything. The model has limited context and limited
attention. Overloaded context degrades performance.

CONTEXT SOURCES (ranked by typical value)
--------------------------------------------------------------------------------

    TIER 1 — always include:
        task_text                   — the actual user request
        linked_issue                — issue body + top comments (if issue URL)
        repo_rules                  — AGENTS.md, CLAUDE.md, .cursor/rules
        repo_map                    — structural overview, file tree, signatures
        build_and_test_commands     — from package.json, Makefile, etc.

    TIER 2 — include if relevant:
        task-adjacent file content  — files mentioned in task or found by search
        recent related PRs          — titles and summaries (if task is a follow-up)
        test configs                — pytest.ini, jest.config.js, etc.
        coding standards            — style guide, architecture conventions
        error logs                  — if task is a bug fix and error is known

    TIER 3 — load on demand (give model tools, don't pre-load):
        full file contents          — model should read specific files as needed
        docs search results         — model should call docs_search when needed
        web search                  — model should call web_search when needed

COLLECTION (node 03 PREPARE_CONTEXT)
--------------------------------------------------------------------------------

    def collect_context(state: AgentState) -> ContextBundle:
        # Always collect
        task_text = state.task_text
        repo_rules = read_repo_rules(state.workspace_path)
        repo_map = get_or_generate_repo_map(state)
        build_commands = detect_build_commands(state.workspace_path)

        # Conditional
        linked_issue = None
        if state.task_source == "issue" or "github.com" in task_text:
            linked_issue = fetch_issue(extract_issue_url(task_text))

        relevant_docs = docs_search(task_text, max_results=3)
        task_files = identify_likely_files(task_text, repo_map)

        return ContextBundle(
            repo_summary=summarize_repo(repo_map),
            task_summary=task_text,
            repo_rules=repo_rules,
            coding_standards=extract_coding_standards(repo_rules),
            build_and_test_commands=build_commands,
            relevant_docs=relevant_docs,
            repo_map=compress_repo_map(repo_map, task_files),
            known_risks=extract_known_risks(repo_rules),
            public_repo_safety_rules=PUBLIC_REPO_RULES if state.is_public_repo else [],
        )

RANKING
--------------------------------------------------------------------------------

Rank context items by:
    - Direct keyword overlap with task_text
    - Files mentioned explicitly in task
    - Files that appear in stack trace (if bug fix)
    - Code ownership metadata (team that owns the files mentioned)
    - Imports / call graph distance from mentioned files
    - Recent changes in the relevant area (git log --since=30days)
    - Test files near mentioned files
    - Doc freshness (prefer recently updated docs)

Implementation:
    - Simple: BM25 text similarity against task_text
    - Better: hybrid BM25 + task-keyword file matching + ownership
    - Best: add embedding similarity for semantic matches

COMPRESSION
--------------------------------------------------------------------------------

Compression strategies for each type:

    Repo map:
        Full tree-sitter map can be huge. Compress by:
        - Include only modules relevant to the task (if identifiable)
        - Show class/function signatures but not bodies
        - Include test file locations (just paths, not content)
        - Cap at 2000 tokens

    File content:
        Don't pre-load files. Give model read_file tool.
        Exception: tiny files directly referenced in task (<50 lines)

    Docs:
        Extract key excerpts, not full pages.
        Each snippet: title, source, updated_at, excerpt (200 tokens max)

    Issue comments:
        Include issue body + top 5 comments.
        Skip bot/automated comments.
        Cap at 1000 tokens.

    Stack traces:
        Include verbatim (they are precise by nature).
        Cap at 50 lines.

CONTEXT FRESHNESS
--------------------------------------------------------------------------------

Every doc snippet carries:
    source: str             # URL or internal path
    retrieved_at: datetime
    updated_at: datetime | null
    confidence: float       # 0-1, relevance score

Stale docs (updated_at > 6 months ago) are flagged in the context bundle
as "potentially outdated."

CONTEXT UPDATES DURING SUBSESSION
--------------------------------------------------------------------------------

As the LLM subsession runs, the context window fills up.
The runtime manages context by:

    - Keeping verbatim: task text, current diff, recent errors, current node
    - Summarizing: old tool outputs that were processed (not failing)
    - Dropping: successful read_file calls where model already acted on content
    - Storing externally: full command outputs (artifact), full file reads

    Context management triggers at 80% of max_context_tokens.
    A summarization pass compresses old tool outputs to 1-2 sentence summaries.

TOKEN BUDGET BY CONTENT TYPE
--------------------------------------------------------------------------------

    Context type            Recommended max tokens
    ─────────────────────────────────────────────
    System prompt           2,000
    Task text               1,000
    Repo rules              1,500
    Repo map                2,000
    Linked issue            1,500
    Relevant docs           1,000
    Build/test commands     500
    Pending user messages   500
    ─────────────────────────────────────────────
    Total pre-call budget   ~10,000

    Remaining context is used for conversation history and tool outputs.
    Most modern models: 128k-200k context window. Use it for history, not
    for front-loading everything — give tools for the rest.
