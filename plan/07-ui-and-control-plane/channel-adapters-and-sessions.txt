================================================================================
CHANNEL ADAPTERS AND SESSION MANAGER
================================================================================

CHANNEL ADAPTERS
--------------------------------------------------------------------------------

Each adapter handles one inbound channel. Adapters normalize to InboundMessage
and delegate all logic to the session manager + agent runtime.

InboundMessage:
    channel: str
    user_id: str
    session_id: str
    content: str
    attachments: list[Attachment]
    reply_fn: callable              # how to send a reply back to this channel
    metadata: dict                  # channel-specific extras

Attachment:
    type: str                       # "image" | "file" | "url"
    url: str | None
    content: bytes | None
    filename: str | None

CLI ADAPTER
    Reads from stdin. Writes to stdout.
    user_id = system username or config file user
    session_id = "cli-main" (or a named session)
    Connects to control plane via WebSocket for live output.

WEB UI ADAPTER
    HTTP POST /api/tasks with JSON body.
    reply_fn = send HTTP response or push via WebSocket to user's browser session.
    user_id from JWT.

SLACK ADAPTER
    Slash command handler: /agent <task>
    Event handler: app_mention, message.im
    reply_fn = post to Slack channel/thread
    user_id = Slack user ID
    Needs: Slack App with Bot Token, slash command config

GITHUB EVENTS ADAPTER
    Webhook receiver for GitHub events.
    Registered at /webhooks/github.
    Verifies HMAC signature.
    Extracts: event type, repo, issue/PR body, user login.
    Maps to InboundMessage with metadata.issue_url or metadata.pr_url.
    reply_fn = post GitHub comment (via github_post_comment tool,
               subject to approval policy)

    Handles: issues.opened, pull_request.*, check_run.completed, push

TELEGRAM ADAPTER
    grammY bot framework.
    Bot command: /task <description>
    reply_fn = sendMessage to chat_id
    user_id = Telegram user ID
    Supports: inline keyboards for approve/deny buttons

WHATSAPP ADAPTER
    Baileys library.
    Trigger: specific contact or group; prefix "agent: <task>"
    reply_fn = sendMessage to JID
    Note: requires self-hosted Baileys instance

SESSION MANAGER
--------------------------------------------------------------------------------

The session manager resolves which session context to use for a message.

Responsibilities:
    - Map (user_id, channel, context) to session_id
    - Create new sessions when none exist
    - Load context files: AGENTS.md, SOUL.md, TOOLS.md, MEMORY.md, DAILY.md
    - Determine if message is a coding task or a conversational turn
    - Pass session context to agent runtime

Session types:
    1. MAIN SESSION
       One per user per channel. Used for conversational turns and status queries.
       session_id = f"main-{channel}-{user_id}"
       Context files in: ~/.agent/sessions/main-{channel}-{user_id}/

    2. TASK SESSION
       One per coding task. Isolated. Created by scheduler on task start.
       session_id = task_id
       Context files in: /workspace/.agent/ (inside sandbox)
       Destroyed with sandbox (artifacts retained)

    3. GROUP SESSION
       One per group chat (for Slack channel, Telegram group).
       session_id = f"group-{channel}-{group_id}"
       Used for shared team agent interactions.

Task vs conversational routing:

    def classify_message(content: str) -> str:
        # Simple heuristics + optional small LLM call for ambiguous cases
        if any(keyword in content.lower() for keyword in TASK_KEYWORDS):
            return "task"
        if content.startswith("/") or "fix" in content.lower():
            return "task"
        return "conversational"

    TASK_KEYWORDS = [
        "implement", "fix", "add", "remove", "refactor", "create", "build",
        "write tests for", "update", "change", "debug", "migrate"
    ]

Context file loading:
    def load_session_context(session_id: str) -> SessionContext:
        base_path = session_path(session_id)
        return SessionContext(
            agents_md=read_if_exists(base_path / "AGENTS.md"),
            soul_md=read_if_exists(base_path / "SOUL.md"),
            tools_md=read_if_exists(base_path / "TOOLS.md"),
            memory_md=read_if_exists(base_path / "MEMORY.md"),
            daily_log=read_if_exists(base_path / "DAILY.md"),
            history=read_if_exists(base_path / "HISTORY.md"),
        )

GOOGLE INTEGRATIONS (v3+)
--------------------------------------------------------------------------------

From README.txt: "google account integrations"

Google Drive tool:
    gdrive_read(file_id_or_query)
        Search or read files from user's Drive.
        Returns: file content (Docs as text, Sheets as CSV snippet, etc.)
        Use case: agent reads a spec doc from Drive as task context.

    gdrive_create(title, content, folder_id?)
        Creates a new Google Doc with agent-generated content.
        Use case: prompt loop generates a report, writes it to Drive.

Google Calendar tool:
    gcal_list_events(timeMin, timeMax)
        Lists events in the user's calendar.
        Use case: scheduling-aware agent loops ("what's blocked this week?")

Gmail tool:
    gmail_read(query, max_results?)
        Searches user's Gmail.
        Use case: agent reads incoming support emails as task triggers.

    gmail_send(to, subject, body)
        DESTRUCTIVE. Requires human approval.
        Use case: prompt loop sends weekly changelog to email list.

All Google tools require OAuth 2.0 authorization.
Authorization flow: user visits /auth/google, grants scopes, token stored.
Tokens are per-user, refreshed automatically, stored encrypted.
Tools are added to "integration" toolset when user has authorized Google.
