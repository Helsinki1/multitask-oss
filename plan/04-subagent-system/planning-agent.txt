================================================================================
PLANNING AGENT
================================================================================

PURPOSE
--------------------------------------------------------------------------------

The planning agent decomposes a complex task into smaller subtasks that
worker agents can execute independently. It runs before any code is written.

The planning agent has NO write access to the repo. It reads, thinks, and
produces a structured subtask list. Nothing more.

WHEN TO USE
--------------------------------------------------------------------------------

Use the planning agent when a task:
    - Requires modifying 5+ files across different subsystems
    - Has clear parallel work streams (e.g. "add feature X to frontend, backend, and tests")
    - Benefits from a dedicated analysis pass before implementation
    - Was explicitly requested as a multi-part job

Do not use for simple tasks. Most tasks should go straight to implementation.
The planning agent adds latency and cost. Default: single-agent implementation.

Heuristic for auto-decision:
    If task_text has > 300 tokens and mentions > 3 distinct subsystems or files:
        spawn planning agent
    Else:
        single IMPLEMENT_TASK subsession

PLANNING AGENT BLUEPRINT (trimmed devloop)
--------------------------------------------------------------------------------

    01 CHECK_FOR_CLEAN_BRANCH
    -> LOAD_PLANNING_TASK
    -> PREPARE_PLANNING_CONTEXT
    -> DECOMPOSE_TASK           (LLM subsession)
    -> VALIDATE_DECOMPOSITION   (deterministic)
    -> FINALIZE_PLAN
    -> END

DECOMPOSE_TASK SUBSESSION
--------------------------------------------------------------------------------

System prompt:
    "You are a software architect planning a coding task.
    Your job is to decompose the task into independent subtasks.
    You have read-only access to the repository.
    Output a structured plan — do not implement anything."

Tools: read_file, search_repo, list_files, get_repo_map, code_search, docs_search

Prompt template:
    "Decompose the following task into subtasks for parallel implementation.

    TASK:
    {task_text}

    REPO MAP:
    {repo_map}

    RELEVANT FILES (from auto-tools):
    {relevant_file_snippets}

    For each subtask, provide:
    - title (short, imperative)
    - description (full implementation instructions, self-contained)
    - files_to_modify (list of specific file paths)
    - files_to_read (list of files to inspect for context)
    - dependencies (IDs of subtasks that must complete first)
    - test_hint (how to verify)

    Rules:
    - Each subtask must be independently implementable
    - Minimize dependencies between subtasks
    - Maximum 6 subtasks — prefer fewer
    - If task is simple enough for one agent, say so and provide single subtask
    - Be specific about file paths — don't say 'the backend files'"

Output schema (structured):
    PlanOutput:
        is_simple: bool     # if true, use single worker, ignore subtasks
        reasoning: str
        subtasks: list[SubtaskSpec]

Max turns: 20
Temperature: 0.3

VALIDATE_DECOMPOSITION (Deterministic)
--------------------------------------------------------------------------------

Checks:
    - All files_to_modify paths exist in repo (git ls-files)
    - No circular dependencies in subtask graph
    - Total subtask count <= max_subtasks (6)
    - Each subtask has a non-empty description
    - is_simple=True or subtasks is non-empty

On validation failure:
    Re-run DECOMPOSE_TASK with error feedback (once)
    If still invalid: fall back to single-agent implementation

PLAN OUTPUT STORAGE
--------------------------------------------------------------------------------

Stored in task DB as:
    - parent task: status = "planning_complete"
    - plan artifacts: SubtaskSpec list as JSON artifact
    - subtask tasks: created and enqueued

After plan is stored, parent task spawns worker agents (see subagent-model.txt).

PLANNING AGENT CONTEXT FILES
--------------------------------------------------------------------------------

The planning agent gets its own context files:
    TASK.md         — full task description
    AGENTS.md       — planning agent capabilities
    REPO_MAP.md     — current repo map

It does NOT get MEMORY.md from other sessions. Plans are fresh each time.
