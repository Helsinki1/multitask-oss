================================================================================
REVIEWER AGENT
================================================================================

PURPOSE
--------------------------------------------------------------------------------

The reviewer agent reads a completed diff (either from a single worker or
from an integration agent) and produces a structured critique before the PR
is submitted for human review.

The reviewer agent has NO write access. It produces a CritiqueReport only.
This report is:
    - Attached to the PR body as a collapsible section
    - Stored as a trace artifact
    - Optionally used by the writer agent to self-correct before final push

WHEN TO SPAWN
--------------------------------------------------------------------------------

Always spawn for:
    - Complex multi-subtask tasks (after integration)
    - Tasks that touch security-sensitive code (auth, payments, permissions)
    - Public repo contributions

Optional for:
    - Single-agent tasks that exceeded 50 turns
    - Tasks where test failures were detected and fixed

Skip for:
    - Simple single-file fixes with passing tests
    - Documentation-only tasks
    - Tasks explicitly flagged as no-review

REVIEWER BLUEPRINT
--------------------------------------------------------------------------------

    LOAD_REVIEW_TASK (D)
    -> PREPARE_REVIEW_CONTEXT (D)
    -> REVIEW_DIFF (L subsession — no write tools)
    -> STRUCTURE_CRITIQUE (D — format and store CritiqueReport)
    -> END

REVIEW_DIFF SUBSESSION
--------------------------------------------------------------------------------

Tools: read_file, search_repo, git_diff, git_log, code_search
No shell execution. No file writes. Read-only only.

System prompt:
    "You are a careful code reviewer. Your job is to identify correctness bugs,
    security issues, and significant quality problems in this diff. You are not
    looking for style nitpicks or minor improvements — focus on things that
    could cause incorrect behavior, security vulnerabilities, or significant
    maintainability problems."

Input:
    - Task description
    - Full diff (or diff artifact reference)
    - Changed files list
    - Validation results from worker/integration agent

Review prompt:
    "Review this diff for:
    1. Correctness bugs (logic errors, off-by-one, null handling, type errors)
    2. Security issues (injection, auth bypass, data exposure, unsafe eval)
    3. Significant missing tests for new behavior
    4. Behavior that doesn't match the task requirements
    5. Calls to APIs or functions that don't exist or are deprecated

    For each finding, state:
    - file and line range
    - what is wrong
    - why it matters
    - suggested fix (brief)

    Do not report style nitpicks. Do not report 'missing comments'.
    Do not report things that are clearly correct but that you'd do differently."

Max turns: 30
Temperature: 0.1
Output schema (structured):

    CritiqueReport:
        overall_verdict: "approve" | "needs_changes" | "major_issues"
        summary: str
        findings: list[CritiqueFinding]
        praise: str                 # what the agent did well (optional)
        not_reviewed: list[str]     # files/areas outside scope of review

    CritiqueFinding:
        severity: "critical" | "high" | "medium" | "low"
        category: "bug" | "security" | "missing_test" | "wrong_behavior" | "bad_api"
        file: str
        line_range: str             # e.g. "42-51"
        description: str
        suggestion: str

USING THE CRITIQUE REPORT
--------------------------------------------------------------------------------

Integration policy:
    - overall_verdict = "approve":
        PR proceeds normally. Critique appended to PR body as detail.

    - overall_verdict = "needs_changes":
        PR is marked draft. Critique findings are listed in PR body.
        Optionally: spawn a FIX_CRITIQUE subsession (re-uses lint-fix pattern)
            Prompt: "The reviewer found the following issues. Fix the high/critical ones.
            Keep changes minimal. Do not fix low/medium findings unless trivial."

    - overall_verdict = "major_issues":
        PR is blocked. User is notified with critique.
        Human must decide whether to proceed.

PR BODY INTEGRATION
--------------------------------------------------------------------------------

    ## Agent Review

    <details>
    <summary>Automated code review</summary>

    **Verdict:** Needs changes

    **Summary:** {critique.summary}

    **Findings:**
    - [HIGH] `src/auth.py:42-51` — {description}
    - [MEDIUM] `src/api.py:103` — {description}

    **Not reviewed:** {not_reviewed}
    </details>
