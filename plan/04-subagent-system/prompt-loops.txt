================================================================================
PROMPT LOOPS — USER-DEFINED RECURRING AGENT BEHAVIORS
================================================================================

WHAT IS A PROMPT LOOP
--------------------------------------------------------------------------------

A prompt loop is a first-class entity that lets users define automated agent
behaviors without writing code. It is not a cron job that runs a shell script.
It is a typed object that, when triggered, submits a full agent task through
the normal devloop pipeline.

Examples:

    "Every time a new issue is opened with label 'bug', run my triage prompt
    and add a severity estimate as a comment."

    "Every Monday morning, run my changelog-generation prompt against main
    since the last release tag, and open a PR."

    "Every time CI fails on main, run my CI-diagnostic prompt and post
    findings to #eng-alerts."

    "After each PR merge, run my test-coverage prompt and open a follow-up
    issue if coverage dropped."

PROMPT LOOP ENTITY
--------------------------------------------------------------------------------

    PromptLoop:
        id: str
        user_id: str
        name: str
        description: str

        # Trigger
        trigger_type: enum      # "cron" | "github_event" | "webhook" | "manual"
        trigger_config: dict    # type-specific config (see below)

        # Prompt
        prompt_template: str    # Jinja2 template, variables from trigger event
        target_repo: str
        task_type: str          # "implement" | "review" | "diagnose" | "generate"

        # Budget
        max_cost_usd: float
        max_wall_seconds: int
        max_runs_per_day: int
        max_concurrent: int     # default: 1

        # Output
        output_policy: enum     # "notify" | "open_pr" | "open_issue" | "post_comment" | "silent"
        notify_channel: str     # channel_id to notify on completion

        # History
        created_at: datetime
        last_run_at: datetime | null
        last_run_status: str | null
        run_count: int
        error_count: int

TRIGGER TYPES
--------------------------------------------------------------------------------

1. CRON TRIGGER
    trigger_config:
        schedule: str           # cron expression: "0 9 * * 1" = Monday 9am
        timezone: str           # "America/New_York"

    Variables available in template:
        {{ today }}             # date
        {{ run_number }}        # sequential run count
        {{ repo }}              # target_repo

2. GITHUB EVENT TRIGGER
    trigger_config:
        event: str              # "issues.opened" | "pull_request.closed" |
                                # "check_run.completed" | "push"
        filters:
            labels: list[str]   # e.g. ["bug"] — only if issue has these labels
            branch: str         # e.g. "main"
            status: str         # e.g. "failure" for CI events

    Variables available in template:
        {{ event.title }}       # issue/PR title
        {{ event.body }}        # issue/PR body
        {{ event.url }}         # URL
        {{ event.author }}      # opener/merger
        {{ event.labels }}      # labels list
        {{ event.branch }}      # branch name
        {{ event.sha }}         # commit SHA (for CI events)
        {{ event.ci_logs }}     # CI failure log excerpt (for check_run events)

3. WEBHOOK TRIGGER
    trigger_config:
        secret: str             # HMAC secret for verification
        payload_schema: dict    # expected JSON structure (optional validation)

    Variables: {{ payload.* }} — any field from verified webhook JSON

4. MANUAL TRIGGER
    No trigger_config. Loop is run on-demand via API or UI.
    Variables: {{ user_input }} — text provided at trigger time

PROMPT TEMPLATE
--------------------------------------------------------------------------------

Prompt templates use Jinja2. They are the full task description passed to the agent.

Example — bug triage loop:

    You are triaging a new bug report.

    Issue: {{ event.title }}
    Body: {{ event.body }}
    URL: {{ event.url }}

    Repository: {{ repo }}

    Tasks:
    1. Read the issue carefully.
    2. Search the codebase for the likely affected code area.
    3. Estimate severity: P0 (production down), P1 (major), P2 (minor), P3 (enhancement).
    4. Identify the likely root cause (1-2 sentences).
    5. Suggest the next step (1 sentence).

    Post your findings as a structured comment on the issue using the issue_comment tool.
    Do not write code. Do not open PRs.

Example — weekly changelog loop:

    Generate a changelog entry for the week ending {{ today }}.

    Repository: {{ repo }}
    Base: last release tag
    Head: main

    1. Run git log since the last release tag.
    2. Group commits by: features, fixes, chores.
    3. Draft a CHANGELOG.md entry in Keep-a-Changelog format.
    4. Open a PR titled "chore: add changelog for week of {{ today }}".

LOOP ENGINE — IMPLEMENTATION
--------------------------------------------------------------------------------

    class PromptLoopEngine:

        def __init__(self, db, scheduler, renderer, tracer):
            ...

        async def handle_cron_tick(self):
            loops = db.get_loops_matching_cron(current_time())
            for loop in loops:
                if self.rate_limit_ok(loop):
                    await self.trigger_loop(loop, event={})

        async def handle_github_event(self, event_type, payload):
            loops = db.get_loops_matching_github_event(event_type, payload)
            for loop in loops:
                if self.matches_filters(loop, payload) and self.rate_limit_ok(loop):
                    await self.trigger_loop(loop, event=payload)

        async def trigger_loop(self, loop: PromptLoop, event: dict):
            rendered_prompt = renderer.render(loop.prompt_template, {
                "event": event,
                "repo": loop.target_repo,
                "today": str(date.today()),
                "run_number": loop.run_count + 1,
            })

            task = Task(
                user_id=loop.user_id,
                repo_url=loop.target_repo,
                task_text=rendered_prompt,
                task_source="loop",
                loop_id=loop.id,
                budgets=BudgetState(
                    max_cost_usd=loop.max_cost_usd,
                    max_wall_seconds=loop.max_wall_seconds,
                ),
                output_policy=loop.output_policy,
                notify_channel=loop.notify_channel,
            )

            task_id = db.insert_task(task)
            loop_run = LoopRun(loop_id=loop.id, task_id=task_id, started_at=now())
            db.insert_loop_run(loop_run)
            scheduler.enqueue(task_id)

            db.update_loop(loop.id, last_run_at=now(), run_count=loop.run_count+1)

        def rate_limit_ok(self, loop: PromptLoop) -> bool:
            runs_today = db.count_loop_runs_today(loop.id)
            active_runs = db.count_active_loop_runs(loop.id)
            return runs_today < loop.max_runs_per_day and active_runs < loop.max_concurrent

LOOP HISTORY + UI
--------------------------------------------------------------------------------

LoopRun entity:
    id: str
    loop_id: str
    task_id: str
    triggered_by: str           # "cron" | "github_event" | "webhook" | "manual"
    trigger_event_summary: str
    started_at: datetime
    ended_at: datetime | null
    status: str
    result_url: str | null      # PR URL, issue URL, etc.
    cost_usd: float
    error: str | null

UI:
    Loop detail page shows:
        - Loop definition
        - Run history (last 100 runs)
        - Per-run: status, cost, duration, result link
        - Enable/disable toggle
        - Edit template
        - Manual trigger button

OUTPUT POLICIES
--------------------------------------------------------------------------------

    "notify":
        Send result summary to notify_channel.
        Do not open PRs or issues from the agent.
        Useful for diagnostic and analysis loops.

    "open_pr":
        Agent runs with allow_push=True.
        After task completes, PR is created normally.
        notify_channel receives PR URL.

    "open_issue":
        After task completes, agent creates a GitHub issue
        with the summary and key findings.
        (Requires issue_create tool, gated by approval policy)

    "post_comment":
        After task completes, agent posts a comment on the
        triggering issue or PR (if event-triggered).
        (Requires issue_comment tool, gated by approval policy)

    "silent":
        Result is stored in DB, no external output.
        Useful for testing loops before enabling output.
