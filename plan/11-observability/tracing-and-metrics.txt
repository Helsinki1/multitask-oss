OBSERVABILITY — TRACING AND METRICS

TRACE MODEL
--------------------------------------------------------------------------------

Every task has a trace: a sequence of typed events covering the full lifecycle.

Events are structured as TraceEvent:
    trace_id: str
    session_id: str
    task_id: str
    timestamp: datetime
    event_type: str
    node_name: str | null
    severity: "debug" | "info" | "warning" | "error"
    message: str
    metadata: dict
    artifact_refs: list[str]

TRACE IMPLEMENTATION (v1 -> v2 evolution)
--------------------------------------------------------------------------------

V1 (MVP):
    - JSONL file per task: one JSON object per line, one event per object
    - Stored in artifact store: /artifacts/{task_id}/trace.jsonl
    - Simple web UI reads JSONL and renders a timeline
    - grep-friendly, diff-friendly

V2:
    - OpenTelemetry spans emitted alongside JSONL
    - Spans exported to: Braintrust, Langfuse, or Phoenix for LLM call inspection
    - Standard spans also exported to Jaeger/Tempo for infrastructure tracing

V3+:
    - Full distributed tracing: parent_task -> subtask spans linked via trace_id
    - LLM-specific tracing: prompt version, model name, latency, cost per call
    - Trace sampling: store 100% for last 7 days, 10% for 90 days
    - Structured search: "all tasks that called tool X in node Y with error Z"

TRACE PAGE (web UI)
--------------------------------------------------------------------------------

The trace page for each task shows:
    - Current status + current node
    - Elapsed time + cost
    - Branch/PR links
    - Timeline of nodes with duration (Gantt-style)
    - For each node: status, duration, key events
    - For each model call: prompt tokens, completion tokens, cost, latency
    - For each tool call: arguments (truncated), result summary, duration
    - Command history (all run_shell / run_tests calls)
    - Validation results (lint, test, policy)
    - Changed files (collapsible diff)
    - Final report
    - Resume / Cancel buttons (if task is still running)
    - Download trace button (JSONL)

CORE METRICS (collect from day 1)
--------------------------------------------------------------------------------

Task outcome metrics:
    task_success_rate           succeeded_pr / total tasks
    pr_created_rate             pr created / total tasks
    branch_only_rate            branch created but no PR / total
    no_change_rate              no files changed / total
    safety_gate_failure_rate    blocked by policy / total
    cancellation_rate
    timeout_rate

Quality metrics (requires human outcome data):
    pr_accepted_rate            human merged without request for changes
    pr_merged_rate
    review_change_request_rate  PR requested changes before merge
    revert_rate
    post_merge_incident_rate

Performance metrics:
    p50/p95/p99 task duration
    p50/p95 cost per task
    p50/p95 model call count per task
    p50/p95 tool call count per task
    tests_run_per_task
    lint_pass_rate
    ci_pass_rate

Subagent metrics:
    subtask_success_rate        by role (planner, worker, reviewer)
    subtask_merge_conflict_rate
    avg_subtasks_per_task

Prompt loop metrics:
    loops_triggered_per_day
    loop_success_rate
    loop_cost_per_run

METRICS IMPLEMENTATION
--------------------------------------------------------------------------------

V1: Simple DB queries over tasks/sessions/validation_results tables.
    Export to CSV or simple charts on the web UI.

V2: Prometheus metrics emitted from agent runner:
    - task_duration_seconds histogram
    - task_cost_usd histogram
    - model_call_latency_ms histogram
    - tool_call_count counter
    Scraped by Prometheus, displayed in Grafana.

V3: Full dashboards:
    - Task success rate over time
    - Cost trends
    - Model usage breakdown
    - Failure category distribution (from taxonomy)
    - Per-repo quality metrics
    - Prompt version A/B comparison

FAILURE TAXONOMY
--------------------------------------------------------------------------------

Classifying why tasks fail helps prioritize improvements:

    bad_task_understanding      model didn't understand what was asked
    bad_context_retrieval       wrong files loaded, missed key context
    wrong_file_edited           plausible but incorrect file
    syntax_or_type_error        code doesn't compile/typecheck
    test_failure                tests fail after implementation
    lint_failure_persistent     lint failures not fixed
    premature_stop              model stopped before completing task
    tool_misuse                 model used tool incorrectly
    sandbox_environment_failure deps missing, services wouldn't start
    credentials_permissions     git push failed, API auth error
    ci_unrelated_flaky          CI failed but not due to this branch
    unsafe_public_leak          safety gate blocked a public repo push
    excessive_cost_time         budget exhausted before completion
    pr_description_hallucination PR claims tests passed that didn't run

Every failed task should be classified. Classification can be:
    Automatic: pattern matching on error messages and trace events
    LLM-assisted: small model reads trace summary, classifies
    Human: reviewer labels production samples
