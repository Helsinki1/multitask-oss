================================================================================
PHASED BUILD PLAN
================================================================================

PERFORMANCE LEVERAGE ORDER (build in this priority order)
--------------------------------------------------------------------------------

1. Strong coding model (already available: claude-opus-4-8)
2. Real executable sandbox (Docker)
3. Excellent file/search/edit/test tools
4. Deterministic validation pipeline
5. Repo rules + context retrieval
6. Done-check + nudge loop
7. CI feedback loop
8. Trace-driven debugging
9. Eval suite
10. Warm sandboxes
11. Subagent system
12. Prompt loops
13. Browser/VLM tools
14. 3D mind palace UI

Do not start with elaborate multi-agent machinery.
Build one robust single agent that can inspect, edit, test, and produce a PR.

PHASE 0 — LOCAL PROTOTYPE (existing: run_agent.py + agent/)
--------------------------------------------------------------------------------

Status: exists (the current codebase)

What it does:
    - Simple LLM loop
    - Basic file/git tools
    - Local repo execution

What it lacks:
    - Deterministic blueprint nodes
    - Sandbox isolation
    - PR creation
    - Trace/observability

This is the foundation. Phase 1 builds on it.

PHASE 1 — MVP: SANDBOXED SINGLE-AGENT WITH PR (Weeks 1-4)
--------------------------------------------------------------------------------

Goal: Given a task + repo, create a branch, implement the change, run tests,
      generate PR. Human can watch it happen.

Build:
    Week 1:
        - Implement AgentState object + state store (SQLite)
        - Implement BlueprintEngine (run nodes in sequence, persist state)
        - Implement nodes 01-05 (branch check, load task, context, implement, checkpoint)
        - Implement core tools: read_file, apply_patch, run_shell, run_tests,
          search_repo, git_diff, git_status, list_files
        - Implement IsDoneChecker (structured output, nudge loop)
        - JSONL trace events
        - Test against demo_repo

    Week 2:
        - Add Docker sandbox (docker-py, per-task container)
        - Add node 06 GATHER_CHANGED_FILES
        - Add node 08 LINT_AND_FORMAT (python: ruff/black/mypy; node: eslint/tsc)
        - Add node 10 POLICY_AND_SECRET_SCAN (gitleaks)
        - Implement auto-tools (repo map, repo instructions reader)
        - Add context bundle + context pipeline (node 03)

    Week 3:
        - Add node 13 PUSH_BRANCH
        - Add node 15 GENERATE_PR_DESCRIPTION
        - Add node 16 CREATE_OR_UPDATE_PR (GitHub API)
        - Add node 19 FINALIZE_AND_NOTIFY (CLI + email)
        - Add nodes 07, 09, 11 (hooks, lint-fix, public repo review)
        - Add CLI adapter + simple status output

    Week 4:
        - Add node 12 RUN_RELATED_TESTS
        - Add WebSocket control plane (basic: broadcast trace events)
        - Add simple web UI (task list, task detail with trace)
        - Add budget enforcement (turns, cost, time)
        - Add cancellation
        - Add fixture repos + Layer 1/2 eval tests

Milestone acceptance:
    - Agent creates a PR for a real bug fix in a test repo
    - Trace page shows every tool call and validation result
    - Cancellation works
    - No host filesystem access

PHASE 2 — TOOL REGISTRY + LOOPS + MODEL GATEWAY (Weeks 5-10)
--------------------------------------------------------------------------------

Goal: MCP-compatible tool registry, approval flow, prompt loops, model gateway.

Build:
    Week 5-6:
        - Implement formal tool registry (ToolSpec, toolsets, sensitivity levels)
        - Implement approval flow (two-phase protocol, web UI approve/deny)
        - Add model gateway (LiteLLM proxy, spend tracking, routing policy)
        - Add docs_search tool (internal docs or web search backend)
        - Add issue_read tool (GitHub API)
        - Add code_search tool (tree-sitter index)

    Week 7-8:
        - Implement prompt loop engine (PromptLoop entity, trigger types: cron + github)
        - Add loop management UI (create, edit, run history)
        - Add GitHub events adapter (webhook handler)
        - Add CI polling (node 17 POLL_CI) + CI-fix subsession (node 18)
        - Add node 09 FIX_LINT_FAILURES LLM subsession
        - Add stronger redaction layer

    Week 9-10:
        - Add Layer 3 golden task eval suite (20 tasks, automated scoring)
        - Add metrics dashboard (basic: task success rate, cost, duration)
        - Add Slack adapter (slash command + interactive approve/deny buttons)
        - Add context freshness tracking (doc updated_at metadata)

Milestone acceptance:
    - Sensitive tools cannot execute without human approval
    - Prompt loops run on schedule and GitHub events
    - Model gateway tracks per-task spend
    - CI failure fix works end-to-end in one test case

PHASE 3 — SUBAGENT SYSTEM + 3D UI (Weeks 11-16)
--------------------------------------------------------------------------------

Goal: Planning, worker, reviewer subagents. Prompt loop webhook triggers.
      3D mind palace TUI.

Build:
    Week 11-12:
        - Implement planning agent (DECOMPOSE_TASK subsession, SubtaskSpec output)
        - Implement work queue + worker blueprint (trimmed devloop)
        - Implement reviewer agent (REVIEW_DIFF subsession, CritiqueReport)
        - Implement integration agent (merge subtask branches, handle conflicts)
        - Add subagent spawning (scheduler + parent-child task linking)

    Week 13-14:
        - Add 3D mind palace TUI (Textual, Python)
            - Floor 1: all active agents grid
            - Room per agent: live log, diff, status
            - Keyboard navigation
            - Approval overlays
        - Connect TUI to control plane WebSocket
        - Add webhook trigger type for prompt loops
        - Add Google Drive tool (gdrive_read, gdrive_create, OAuth flow)

    Week 15-16:
        - Warm sandbox pool (N pre-cloned containers per repo)
        - Add production trace review workflow (sample 5%, human rating UI)
        - Add failure taxonomy auto-classification
        - Regression workflow: compare eval suite against baseline on deploy

Milestone acceptance:
    - Complex task decomposed into 3 subtasks, all implemented in parallel
    - Mind palace shows all active tasks spatially
    - Warm sandbox start time < 30 seconds
    - Eval regression suite runs in CI

PHASE 4 — DISCOVERY TOOLS + FULL MULTI-CHANNEL (Weeks 17+)
--------------------------------------------------------------------------------

Goal: Web search, browser automation, VLM tools, full multi-channel support.

Build:
    - local_server_start/stop/request tools
    - browser_open, browser_screenshot, browser_click, browser_type
    - vlm_describe_screenshot, vlm_compare_screenshots
    - web_search, web_fetch
    - Telegram adapter (grammY)
    - WhatsApp adapter (Baileys)
    - Gmail tool (with OAuth, approval-gated send)
    - Google Calendar tool
    - Firecracker/Kata microVM sandbox option
    - Multi-region sandbox pool
    - Full eval suite expansion (UI task tests)
    - tmux backend for 3D TUI (real terminal panes)

WHAT TO LEAVE OUT OF V1
--------------------------------------------------------------------------------

    - Persistent vector memory (nice to have but not critical)
    - Many cooperating agents (v3+)
    - Production deploy tools (too risky without more safety infrastructure)
    - Automatic PR merging (always keep human in the loop)
    - Broad internal tool access (MCP tools per org — scope carefully)
    - Complex semantic search before grep and repo map
    - Custom model training / fine-tuning
    - Browser automation (v4+)

The v1 milestone is not "replace engineers."
It is "take a well-scoped issue, create a reasonable PR, run relevant checks,
and produce a trustworthy draft — fast."
