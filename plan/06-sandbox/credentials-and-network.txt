================================================================================
SANDBOX CREDENTIALS AND NETWORK POLICY
================================================================================

CREDENTIAL DESIGN
--------------------------------------------------------------------------------

The agent should never see production secrets. Every credential is:
    - task-scoped: only the minimum needed for that task
    - short-lived: expires after task_duration + 1 hour
    - injected at sandbox start: not baked into images
    - redacted from logs: automatically removed before storage

Credentials provided per task:

    GITHUB_TOKEN
        Scope: repo read + single-repo push (not all repos)
        Created via: GitHub fine-grained personal access token
                     OR GitHub App installation token
        Preferred: GitHub App per organization (revocable, auditable)
        Expiry: 1 hour after task expected completion

    ARTIFACT_UPLOAD_CREDENTIALS
        Scope: write to /artifacts/{task_id}/ prefix only
        Service: S3 presigned URL or GCS signed URL
        Expiry: 4 hours

    MODEL_GATEWAY_TOKEN
        Scope: call model gateway with task_id metadata
        Used for spend tracking, rate limits per task
        Not a provider API key (gateway handles that)

What is NOT provided:
    - Production database credentials
    - Production API keys
    - Admin tokens for any service
    - Credentials from .env files or secrets managers
    - SSH keys to production hosts

NETWORK POLICY
--------------------------------------------------------------------------------

Default allowlist (whitelist approach):

    ALLOWED:
        npm registry (registry.npmjs.org)
        PyPI (pypi.org)
        crates.io
        pkg.go.dev
        rubygems.org
        Maven Central
        GitHub (api.github.com, github.com)
        GitLab (api.gitlab.com, gitlab.com)
        Model gateway (internal)
        Artifact store (internal S3/GCS)
        Internal docs service
        Internal code search service

    DENIED (explicitly blocked or not in allowlist):
        Production databases (RDS, Cloud SQL hostnames)
        Internal admin APIs
        Metadata endpoints (169.254.169.254 — AWS IMDSv1)
        All other internal hostnames not on allowlist

Network policy implementation options:
    - Docker: custom bridge network + iptables rules
    - Kubernetes: NetworkPolicy resources
    - VM: security group / VPC firewall rules

Egress logging:
    All outbound connections logged with:
        timestamp, destination_ip, destination_host, bytes_sent, bytes_received
    Logs stored as artifact. Reviewed automatically by policy scanner.

CREDENTIAL INJECTION
--------------------------------------------------------------------------------

At sandbox start, the orchestrator injects credentials via:
    - Environment variables (for simple token values)
    - Mounted secret volumes (for key files)
    - Never baked into images

Injection pseudocode:

    def start_sandbox(task: Task) -> SandboxHandle:
        token = github_app.create_installation_token(
            repo=task.repo_url,
            permissions=["contents:write", "pull_requests:write"],
            expires_in=seconds(task.max_wall_seconds + 3600)
        )

        artifact_creds = artifact_store.create_task_credentials(task.task_id)
        model_token = model_gateway.create_task_token(task.task_id, task.user_id)

        env = {
            "GITHUB_TOKEN": token.value,
            "REPO_URL": task.repo_url,
            "TASK_ID": task.task_id,
            "BRANCH_NAME": task.working_branch,
            "ARTIFACT_ENDPOINT": artifact_creds.endpoint,
            "ARTIFACT_TOKEN": artifact_creds.token,
            "MODEL_GATEWAY_URL": config.model_gateway_url,
            "MODEL_GATEWAY_TOKEN": model_token,
        }

        return docker_backend.start(image=BASE_IMAGE, env=env, ...)

CREDENTIAL REDACTION
--------------------------------------------------------------------------------

Before any log line, command output, or artifact is stored:
    - Run redaction pass: replace known token patterns with [REDACTED]
    - Patterns: GITHUB_TOKEN, ARTIFACT_TOKEN, MODEL_GATEWAY_TOKEN values
    - Regex: common secret patterns (Bearer tokens, AWS key IDs, etc.)
    - Never store raw environment variables in trace

The redaction pass runs in the tool execution wrapper, before output
is stored as an artifact or sent to the model.

PUBLIC REPO PUSH CREDENTIAL ISOLATION
--------------------------------------------------------------------------------

For tasks targeting public repos, an additional isolation layer:

1. At sandbox start: configure git remote push URL to a local bare repo
       git remote set-url --push origin /tmp/agent-local-bare-{task_id}.git
       git init --bare /tmp/agent-local-bare-{task_id}.git

2. Model cannot push to the real remote during implementation.
   git push goes to the local bare repo.

3. After PUBLIC_REPO_REVIEW passes in node 11:
   Deterministic code restores the real remote URL:
       git remote set-url --push origin {real_remote_url}
   Then PUSH_BRANCH (node 13) pushes to real remote.

4. If PUBLIC_REPO_REVIEW fails:
   Real remote URL is never restored. Branch stays local.
   User gets link to local branch and instructions for manual review.
