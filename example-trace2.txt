SYSTEM ARCHITECTURE: END-TO-END FLOW
==============================================================================

ENTRY POINT
-----------
swe_bench_run.py loads a SWE-bench instance, clones the target repo at
base_commit, applies and commits test_patch (the failing tests), pip installs
the repo, then calls run_agent() → build_devloop() → BlueprintEngine.run().

NODE GRAPH
----------
BlueprintEngine is a simple state machine. Each node returns a NodeResult
with {next_node, state_update, status}. The engine loops until next_node=="END".
Exceptions in a node route to that node's failure_next.

  01_CHECK_BRANCH
       ↓
  SEED_SCRIPTS
       ↓
  02_LOAD_TASK
       ↓
  03_PREPARE_CONTEXT
       ↓
  REPRODUCE_ISSUE
       ↓
  04_IMPLEMENT_TASK ←────────────┐
       ↓                         │ (up to 3x)
  VERIFY_FIX ────────────────────┘
       ↓ (after 3 failures, or on pass)
  05_CHECKPOINT
       ↓
  END


WHAT EACH NODE DOES
--------------------
01_CHECK_BRANCH (deterministic)
  - git rev-parse HEAD (confirms git repo)
  - git status --porcelain (aborts if uncommitted changes exist)
  - If on protected branch (main/master/etc), creates agent/<id>-<slug> branch
  - Records initial_commit_sha, working_branch

SEED_SCRIPTS (deterministic)
  - Writes _agent_scripts/docker_run.py into the workspace
    (a helper script to exec commands inside a Docker container if needed)
  - Adds _agent_scripts/ to .gitignore

02_LOAD_TASK (deterministic)
  - Validates task_text is non-empty
  - Derives repo_name from workspace path basename
  - Sets is_public_repo=False (Phase 2 placeholder)

03_PREPARE_CONTEXT (deterministic + optional 1 cheap LLM call)
  - Calls build_context_bundle(): builds a repo map + task-adjacent files
  - In eval_mode (SWE-bench): skips classifier, always routes → REPRODUCE_ISSUE
  - In normal mode: one cheap LLM call classifies task as bug_fix or additive,
    routes to REPRODUCE_ISSUE (bug_fix) or directly to 04_IMPLEMENT_TASK (additive)

REPRODUCE_ISSUE (llm_subsession in normal mode / deterministic in eval_mode)
  - In eval_mode: runs `pytest {FAIL_TO_PASS tests}` and confirms exit code != 0.
    No LLM involved. Just verifies the bugs are actually present before the agent starts.
  - In normal mode: runs an LLM subsession (up to repro_max_turns=15) that builds
    _repro_env/run_repro.py and confirms it exits non-zero with the matching error.
    If first attempt fails, runs a correction subsession (up to 5 turns).
    Sets repro_confirmed, repro_output in state.

04_IMPLEMENT_TASK (llm_subsession — the main loop)
  - Builds system prompt from 7 prompt layers: base constraints, env/branch, tools
    reference, bash skills, repo rules/standards, repro context, verify failures
  - Passes the task text + pre-loaded relevant files as the human message
  - Calls run_subsession() — see SUBSESSION LOOP below
  - Always routes to VERIFY_FIX (status=ok if done, warning if not)

VERIFY_FIX (deterministic — the gate)
  - In eval_mode:
      runs pytest FAIL_TO_PASS tests → must exit 0
      runs pytest PASS_TO_PASS tests → must exit 0 (regression check)
  - In normal mode:
      runs _repro_test.py / _repro_env/run_repro.py → must exit 0
      runs build_and_test_commands from context_bundle
  - On pass → 05_CHECKPOINT
  - On fail → back to 04_IMPLEMENT_TASK (with failure output injected into next
    system prompt via LAYER_VERIFY_FAILURE_TEMPLATE)
  - After MAX_VERIFY_ATTEMPTS=3 failures → force 05_CHECKPOINT (avoid deadlock)

05_CHECKPOINT (deterministic)
  - git add -A
  - git commit -m "agent: implement task '<first 60 chars>'"
  - Records new HEAD SHA
  - Routes → END


THE SUBSESSION LOOP (run_subsession in agent/subsession.py)
------------------------------------------------------------
This is the core LLM turn cycle inside any node that needs an LLM:

  messages = [system_prompt, initial_human_message]
  for turn in range(max_turns):
    1. Check budget (cost_usd, wall_seconds) → exit if exhausted
    2. Check escalation (switch to stronger model if stuck)
    3. If turn >= 20: compress stale tool result messages in history
       (truncates tool outputs beyond last 6 turns to 120 chars each)
    4. Call OpenAI chat completions API with messages + tools schema
       (temperature=0, parallel_tool_calls=False)
    5. Record token cost, update state.budgets
    6a. If finish_reason == "tool_calls":
          parse tool name + args from response
          call ToolRegistry.execute(name, args, workspace)
          append tool result to messages as role="tool"
          continue loop
    6b. If finish_reason == "stop" (no tool call):
          call check_is_done(task_text, last_message, state)  ← another LLM call
          if is_done=True → return status="done"
          else → append NUDGE_TEMPLATE to messages, continue loop
  → if loop exhausts max_turns → return status="budget_exhausted"
  → if openai.APIError → return status="error"

Result status propagates to node: "done"→ok, anything else→warning.


TOOL CALLING MECHANISM
-----------------------
Tools are passed to the OpenAI API as the `tools` parameter (OpenAI function
calling format). When the model wants to use a tool it returns:
  finish_reason = "tool_calls"
  message.tool_calls = [{id, function: {name, arguments (JSON string)}}]

The subsession loop dispatches to ToolRegistry.execute(name, args, workspace),
which calls the registered Python function. Result is a string appended as:
  {"role": "tool", "tool_call_id": tc.id, "content": result_str}

The model sees the result on the next turn.


TOOLS AVAILABLE (build_dev_toolset — 4 tools)
----------------------------------------------
run_shell
  - Runs any bash command in the repo root (or specified cwd)
  - Returns exit_code + stdout + stderr
  - Used for: grep, find, git, pytest, pip install, python scripts
  - This is the workhorse — exploration, tests, installs all go through here

read_file
  - Reads a file with line numbers
  - Accepts start_line/end_line to slice large files
  - Returns numbered lines

write_file
  - Creates a new file (not for overwriting)

replace_in_file
  - Exact-text find-and-replace in an existing file
  - Fails if old_text not found (prevents silent misses)
  - expected_replacements defaults to 1

ROOT CAUSE OF AGENT FAILURE
-----------------------------
The agent correctly identified that __slots__ = () needed to be added, but
Python's __slots__ suppression of __dict__ requires EVERY class in the MRO
to declare __slots__ = (). Sympy's class chain is:
  Symbol → AtomicExpr → Atom → Basic → ... (+ mixins via StdFactKB)

If any one ancestor is missing __slots__ = (), the instance still gets __dict__.
The agent kept patching Symbol and Basic in isolation but never traced the full
MRO or found which specific ancestor was responsible for the regression.
It also had no ability to run a full MRO dump early and reason about all classes
at once — it read them one at a time and lost track across subsessions since each
subsession starts with a fresh message history (no memory of prior iterations).
